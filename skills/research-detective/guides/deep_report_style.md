# 深度分析报告叙事风格

> 本文件是深度分析报告的**默认表达 profile**。写完整/正式/深度研究报告时加载，用来把分析产物转成可阅读、可判断、可交付的最终报告，而不是研究模板填空。
>
> 它不替代 [writing_style.md](writing_style.md) 和 [report_principles.md](report_principles.md)：前者管句法和 AI 味红线，后者管结构和论证质量。本文件只管**深度报告如何组织叙事、建立阅读节奏、呈现发现和给出建议**。
>
> 使用优先级：事实与证据 > `report_principles.md` 的结构和论证质量 > `writing_style.md` 的措辞红线 > 本文件的叙事风格。若发生冲突，不得为了读感牺牲事实、证据强度、可追溯性或红线。

---

## 一句话

> **先给判断，再展示证据；让读者快速知道答案，也看见答案是怎么被推出来的。**

---

## 适用场景

适合：

- 用户要求「深度分析报告」「正式报告」「完整报告」「给业务/客户看的报告」
- 报告需要支持产品、市场、策略、设计、运营或研究决策
- 资料来自访谈、问卷、用户反馈、舆情、竞品、文献等多源材料
- 读者既需要快速拿到结论，也需要相信结论的推导过程

不适合：

- 只需要 300-500 字研究简报
- 只做材料归档或原始资料整理
- 只要学术论文式方法说明
- 证据不足以形成判断，只能列出问题清单

---

## AI 落笔流程

加载本文件后，按下面顺序写最终报告：

1. **定中心判断**：从研究问题和证据链中提炼 1 句中心论点。提炼不出时，回到分析阶段，不要硬写报告。
2. **排读者路径**：决定读者先看什么才能理解判断。常用路径是：问题意识 → 核心答案 → 全局地图 → 关键故事 → 机制解释 → 决策含义。
3. **选章节骨架**：仍以 [templates/simple_report.md](../templates/simple_report.md) 为骨架，但正文标题和段落节奏按本文件调整。
4. **写证据层**：每个关键发现至少挂 3 类材料中的 2 类：原话 / 行为或场景 / 数据或外部资料。证据不足时，把发现降级为假设。
5. **做读感修订**：删除不支撑中心判断的材料，补上故事、表格、边界、反例和 so what。

落笔时不要把本文件的章节标题原样写进报告。它是写作控制台，不是报告目录。

---

## 核心读感

这类报告的好读感来自一种张力：**先用清晰判断抓住读者，再用研究过程让读者相信判断不是拍脑袋。** 它可以有强标题、强开头和强收束，但中间必须不断回到样本、原话、数据口径、反例和边界。

最终读感应当是：专业、有判断、有证据、有温度；不像公众号，不像咨询黑话，也不像学术论文。

## 推荐叙事结构

### 1. 标题：一句话判断，不是话题名

标题承担第一层结论，不只承担命名功能。读者扫标题时，应能知道报告的核心判断、变化方向或关键矛盾。

标题倾向：

- 用判断句，而不是名词短语
- 指向变化、机制、阻力或机会
- 让读者知道「这份报告发现了什么」

避免：

- 只写研究对象，如「用户需求研究报告」
- 只写材料类型，如「XX 调研总结」
- 使用空泛概念，如「深度洞察」「趋势分析」
- 凑「三大」「五大」这类整齐标题

### 2. 开头：用 3 句话完成进入

开头不要铺行业背景。建议用 3 句话完成：

1. **具体问题**：这份报告要解决什么真实决策问题。
2. **核心判断**：当前证据支持什么答案。
3. **阅读承诺**：下面会从哪些证据和分析路径展开。

如果上下文里没人提出过对立观点，不要硬写「不是 X，而是 Y」。直接写判断。

### 3. 执行摘要：先给答案，再给证据入口

执行摘要控制在 1 页内，包含：

- **中心论点**：一句话回答研究问题
- **关键发现**：3-5 条，每条是判断句
- **证据强度**：强 / 中 / 弱，说明口径
- **决策含义**：每条发现对读者意味着什么
- **最大不确定性**：最可能推翻当前判断的证据缺口

摘要不要堆原话和脚注。原话、过程和细节下沉到正文。

### 4. 主体：地图、故事、机制、判断

主体章节不要一路平铺发现。优先采用这种节奏：

1. **地图**：先用分群、路径、场景或数据表说明全局结构。
2. **故事**：再用 1-2 个具体用户、原话或场景让结构变得可感。
3. **机制**：解释故事背后的共同驱动、阻力或矛盾。
4. **判断**：收束到对产品、品牌、市场、设计或后续研究的含义。

每个关键发现都要包含「现象 → 机制 → 证据 → 边界 → so what」。读者应能看到推理链，而不是看到全部工作痕迹。

### 5. 方法说明：短，但要让人信

方法说明不单独写成理论课。只交代读者判断结论强度所需的信息：

- 资料来自哪里，覆盖了哪些人/场景/时间
- 使用了哪些分析动作
- 样本有什么偏差
- 哪些问题没有被材料覆盖

避免用「科学严谨」「多维度」「深入洞察」替代具体方法；也不要只说采用某框架，而不说明这个框架如何影响结论。

### 6. 建议与结尾：收束到决策

建议不是「加强」「优化」「提升」这类动作词堆叠。每条建议都必须包含：

| 要素 | 要回答的问题 |
| --- | --- |
| 动作 | 具体建议做什么 |
| 证据锚 | 基于哪些观察、原话或数据 |
| 适用边界 | 对哪类用户、场景或阶段成立 |
| 验证方式 | 下一步看什么指标或证据判断它是否成立 |

结尾回答三件事：

- 当前最可信的判断是什么
- 读者现在可以做什么
- 哪些判断必须等补证后再做

不要用「综上所述」「值得深思」「未来可期」收尾。

## 表达原则

### 1. 像研究员对同行和业务方说话

可以用「我们观察到」「当前证据支持」「我们判断」。

避免：

- 过度权威腔：「研究表明」「显而易见」「毋庸置疑」
- 公众号腔：「你有没有发现」「真正厉害的人」
- 咨询黑话：「重构链路」「价值跃迁」「打造闭环」

### 2. 具体优先于完整

可读性优先于资料全量展示。删掉不支撑中心判断的数据。

- 一个章节只服务一个关键发现
- 一个段落只讲一个判断
- 证据 3-5 条够用，超过 7 条说明还没归纳
- 有趣但不影响决策的发现放附录

### 3. 图表为判断服务

每张图表前后都要有解释：

- 图表要说明什么判断
- 读者应该看哪一列/哪一处
- 这个图表对决策意味着什么

优先使用：核心发现摘要表、用户旅程 / 行为路径图、分群对比表、证据强度表、反例与边界表、建议优先级表。不要为了「看起来像报告」硬塞图。

---

## 信息密度与排版

- 段落不超过 4 行
- 每 300-500 字至少有一个视觉锚点：表格、引用、数字、路径图或小结框
- 重要数字单独呈现或加粗，但不要整段加粗
- 摘要少放引用，正文再展开证据
- 表格列数控制在 5 列以内
- 长附录和原始材料索引放文末，不打断主线

---

## 交付判定

一份报告只有同时满足下面三类条件，才算使用了本 profile：

1. **读者能快速拿到答案**：标题、开头和执行摘要已经形成完整判断，不依赖读完整正文才能知道结论。
2. **读者能相信答案**：关键判断旁边有证据口径、样本边界、反例或不确定性；强判断不靠单条故事支撑。
3. **读者知道下一步做什么**：建议不是泛泛方向，而是带证据锚、适用边界和验证方式的行动假设。

如果只满足第 1 条，报告会像观点文章；只满足第 2 条，报告会像研究备忘录；只满足第 3 条，报告会像咨询建议清单。深度分析报告必须三者同时成立。

---

## 红线

1. 不出现参考来源痕迹：不得出现公众号、媒体、作者、机构名等风格来源信息。
2. 不只写现象，不写判断：每节都必须回答「所以呢」。
3. 不只给观点，不给证据：核心判断必须能回到原始材料或证据链。
4. 不用模板化标题：同级标题不能长得像同一个句式填空。
5. 不用大词包装洞察：抽象词必须翻译成具体人、场景、动作。
6. 不把建议写成口号：建议必须有证据锚、边界和验证方式。
7. 不隐藏反例：每个核心发现至少说明一个边界、反例或不确定性。
8. 不为了完整牺牲可读性：不支撑主判断的材料下沉附录。

## 自检清单

落笔后逐条检查：

- [ ] 标题是一句话判断，不是话题名
- [ ] 开头 200 字内讲清问题、答案和展开路径
- [ ] 执行摘要控制在 1 页内，先给结论，不堆原话
- [ ] 主体章节遵循「地图 → 故事 → 机制 → 判断」的节奏
- [ ] 每个关键发现都有「现象 → 机制 → 证据 → 边界 → so what」
- [ ] 方法说明足以判断证据强度，但没有写成方法论展示
- [ ] 每条建议都有证据锚、适用边界和验证方式
- [ ] 反例、样本偏差和不确定性没有只放在附录
- [ ] 图表都服务于判断，不是装饰
- [ ] 全文没有出现公众号、媒体、作者、机构名等参考来源痕迹
