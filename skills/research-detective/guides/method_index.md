# 侦探方法入口（Method Index）

> 本文件是 `guides/` 的方法入口页。正式分析前先读这里，再根据课题类型到 `detective_toolkit.md` 末尾速查表选择工具。不要把 26 个工具默认全跑。

## 1. 侦探方法论解决什么

侦探方法论不是替代访谈分析、问卷统计、竞品研究或文献综述，而是在这些成熟方法之上增加一层元分析：

- 防止读到后面忘了前面。
- 防止只看高频主题，忽略低频高风险信号。
- 防止把“说得通的故事”误当成事实。
- 防止结论和证据之间断链。
- 防止报告写完后没人主动找反证。

## 2. 五个侦探动作

| 动作 | 产物 | 目的 |
| --- | --- | --- |
| 3a 全量记忆编码 | `process/3a_coding.md` | 系统记录观察、引用、行为/态度、痛点、积极信号和上下文 |
| 3b 盲区扫描 | `process/3b_blind_spots.md` | 找低频高强度、沉默信号、应出现但缺失的证据 |
| 3c 全局关联发现 | `process/3c_associations.md` | 找跨主题、跨人群、跨数据源的隐藏关联 |
| 3d 矛盾审计 | `process/3d_contradictions_audit.md` | 给每个核心结论找反面证据和替代解释 |
| 3e 证据链追溯 | `process/3e_evidence_chains.md` | 列支持/反对证据、计数、置信度和边界 |

补充标准见 [`research_methodology.md`](research_methodology.md)。

## 3. 26 工具箱如何参与

`detective_toolkit.md` 是专业方法工具箱，不是固定流程清单。正式分析前必须做一次方法选择：

1. 判断课题类型：用户需求探索 / 竞品分析 / 设计方案验证 / 战略方向判断 / 数据摸底 / 因果机制研究 / 体验优化 / 产品定位。
2. 查 [`detective_toolkit.md`](detective_toolkit.md) 末尾速查表。
3. 选择本案需要的采集层、结构层、判断层、综合层工具。
4. 写入 `process/0_method_selection.md`。
5. 只执行选中的工具；未选工具要简要说明为什么不选。

## 4. `process/0_method_selection.md` 格式

```md
# 本次方法选择

## 研究类型

- 类型：用户需求探索 / 竞品分析 / 设计方案验证 / ...
- 判断理由：为什么本案属于这个类型

## 选用工具

| 层级 | 工具 | 用在本案哪里 |
| --- | --- | --- |
| 采集层 | 工具 1：线索提取 | 从访谈/问卷/wiki 中分拣事实、情感、矛盾、异常、沉默线索 |
| 判断层 | 工具 22：Linchpin 检查 | 找核心结论最脆弱前提 |

## 未选工具及理由

- 工具 25：情景规划 —— 本案不是高不确定性的战略方向判断。
- 工具 10：扎根编码 —— 已有明确研究问题，不是全新领域探索。

## 方法风险

- 证据缺口：哪些问题材料不足。
- 偏误风险：最可能出现的确认偏误、锚定或幸存者偏误。
- 下一步验证：哪个最小验证能降低判断风险。
```

## 5. 输出里如何显性体现侦探层

- **研究简报**：必须至少包含一条“反证/边界/待验证”信息。
- **证据链图谱**：每个核心结论要有支持证据、反面证据或“已搜未找到”。
- **深度分析报告**：侦探备忘录可以作为独立章节，也可以分散在主文，但分析必须实际完成。
- **AI 接力包**：必须保留负面清单、未解决问题和证据 ID，防止下游 AI 脑补。

## 6. 什么时候不用重型工具

以下情况不要强行完整跑 26 工具：

- 用户只要少量材料初看：应转轻量阅读或后续 scout，而不是完整 detective。
- 用户只是做报告对抗审查：转 `research-reviewer`。
- 用户只是资料入库：转 `research-archivist`。
- 用户只问研究方法设计：转 planner 或普通研究规划，不进入 detective。
