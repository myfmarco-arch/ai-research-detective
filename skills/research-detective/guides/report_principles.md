# 研究报告规范

> 本文件管**报告的结构与论证质量**。配套文件 [writing_style.md](writing_style.md) 管**句子和措辞**(去 AI 味、概念癌、套路句式)。
> 两份合起来构成研究报告的完整产出标准。模板([templates/](../templates/))只放骨架,判据在这里。

## 适用时机

- 落笔前:作为生成约束通读一遍,把规范装进上下文
- 落笔后:用文末的 13 项自检清单逐条勾选

## 三层原则

报告质量由三层组成,**底线 → 结构 → 表达**。冲突时优先级一致——底线永远高于结构,结构永远高于表达。

```
┌─────────────────────────────────────────────┐
│  底线层(不达标 = 报告作废)                     │
│  · 尊重事实                                    │
│  · 直击目标                                    │
├─────────────────────────────────────────────┤
│  结构层(金字塔原理)                           │
│  · 结论先行                                    │
│  · 自上而下分层                                │
│  · 横向 MECE                                  │
│  · 纵向 Q&A 链                                │
│  · 归纳为主                                    │
│  · 概括必须"实"                               │
│  · so what 三连                               │
├─────────────────────────────────────────────┤
│  表达层(让读者能读、想读、读懂)                │
│  · 有洞察、有观点                              │
│  · 易于理解                                    │
│  · 可视化与美观                                │
│  · 引人入胜                                    │
└─────────────────────────────────────────────┘
```

## 第一层 底线

### 1.1 尊重事实

**核心**:数据和判断必须可追溯、可验证、不过度解读。

约束:

- 每个数据点标具体来源(访谈编号 / 问卷题号 / 第三方报告 + 日期)
- 每个结论标证据强度:强(多源交叉,≥3 个独立来源且类型不同)/ 中(同类多源,≥3 条独立访谈一致)/ 弱(单条 / 单次观察 / 间接推断)
- **n<10 的分群禁用百分比**,改用绝对数 + 备注"信号性观察,样本不足以推断分布"
- **宽口径 vs 窄口径数据必须分开**——例:AI 剪辑提及率拆成"主动强诉求 X%"和"被问到时的认可 Y%",不混算成一个数
- 已知样本偏差(如某品牌过采样)在**受影响的发现旁**用 ⚠ 标注,不只放附录免责
- 每个核心结论主动列**反面证据 / 边界条件 / 在什么情况下不成立**,不留给 reviewer 兜底
- "深刻"不等于"深层"。用户能直接说出来的痛点是**中层**;**深层**是用户否认或没意识到的(例:口头说为记录孩子,实际录完不看)

自检:把报告里所有 "%""倍数""排名""增长" 挑出来,每个都问:**这个数怎么算的?谁能复现?**——复现不了的删掉。

### 1.2 直击目标

**核心**:报告每一节的存在理由是回应"研究要支持的业务决策"。

约束:

- 研究背景章节必须明确写出**这份报告要支持的具体决策**(不是"了解市场",是"决定是否进入 X 品类 / 是否上 Y 功能 / 是否调整定价")
- 中心论点必须**直接回答**研究目标提出的问题
- 每个发现末尾必须有 **"so what / 对决策意味着什么"** 一行
- 不在研究目标范围内的"有趣发现"放附录,不进主线

自检:把报告所有章节标题列出来,问:**删掉这一章,业务决策会受影响吗?**——不会,挪附录或删掉。

## 第二层 结构(金字塔原理)

### 2.1 结论先行(SCQA + Answer)

每一份文档/章节起手按 **情境(Situation)→ 冲突(Complication)→ 疑问(Question)→ 答案(Answer)** 展开,**先给答案,再给支撑**。

应用:

- 报告开头 200 字内完成 SCQA,让读者立刻知道"研究要回答什么 + 答案是什么"
- 核心结论紧跟"研究背景"之后,在"详细分析"**之前**——读者 1 分钟内拿到 takeaway
- 不允许"铺垫四章再揭晓"

### 2.2 自上而下分层

报告是倒树:

```
        中心论点(1 句话)
       /        |        \
   关键论点 1  关键论点 2  关键论点 3
   /   |   \    /  |  \    /  |  \
  证据 证据 证据 ...
```

每一层都在回答"上一层为什么成立"。

约束:

- **中心论点只有 1 个**
- 关键论点 3-7 个(对应详细分析的章节)
- 每个关键论点下挂 3-5 条证据
- 出现"第五章才是真正核心结论"= 结构倒了,要重组

### 2.3 横向 MECE

同一层的并列项必须**一刀切到底**——同一个分类维度,互不重叠、合起来完整。

约束:

- 详细分析的章节必须按**同一维度**展开(选一种:按需求 / 按用户群 / 按场景 / 按决策点)
- 横切维度(人群、地域、时间)作为**交叉分析**穿插进每个主题,**不另开章节**
- 同一条证据/原话不应出现在多个章节(出现 = 维度混了)

MECE 检查(写完详细分析的章节标题后必做):

1. 这些章节是不是按**同一个维度**切的?
2. 同一条证据有没有出现在多个章节?
3. 合起来覆盖了研究目标吗?有没有遗漏?

任一条不过 → 重切。

### 2.4 纵向 Q&A 链

每个标题/小节存在的理由,是回答上一层标题自然引发的"为什么 / 怎么做到 / 凭什么"。

自检:读章节目录,每个标题问自己——**"读完上一层,读者会自然问出这个问题吗?"**

- 读完"研究背景" → 读者问"那答案呢?" → 下一节是"核心结论" ✅
- 读完"核心结论的关键论点 1" → 读者问"凭什么这么说?" → 下面是"详细分析—主题 1" ✅
- 读完"详细分析" → 读者问"那然后呢?" → 应当有"待解问题/后续研究"

### 2.5 归纳为主

研究报告主体使用**归纳论证**:并列证据 → 共同结论。结论强度 = 证据归纳的可靠程度。

约束:

- 每个发现是"论点 + N 条平行证据 + 共同归纳"的结构
- 演绎论证(大前提 → 小前提 → 结论)只在论证关键判断时用,不做主结构
- 不混用——一个章节里不要前半归纳后半演绎

### 2.6 概括必须"实"

中心论点和关键论点不能是概念性总结,必须**具体到能让读者做判断或行动**。

| 概念型(❌) | 实型(✅) |
|------------|------------|
| 运动相机面临转型挑战 | 运动相机的购买驱动已从极限运动场景迁移到日常记录场景,要求产品功能优先级重排 |
| 用户分群 | 购买驱动横跨 6 类用户,其中最大群体并非运动用户 |
| 三层需求结构 | 用户的真实痛点是续航,但深层驱动是怕忘记 |

约束:

- 每个章节标题独立成句、传达判断
- 读者扫一遍目录就知道结论
- 不允许 "不是 X,而是 Y" 句式做标题(口号化、削弱具体性)

### 2.7 so what 三连

每写一段证据,问自己三遍 **"所以呢?"** ——直到追问到读者真正在乎的层面(决策 / 行动 / 风险)。

约束:

- 每个发现末尾必须有"so what / 决策含义"一行
- 追问到第 3 层(意味着什么 → 影响谁 → 该做什么),不到 = 决策含义没写够

例:

```
事实:70% 用户提及续航焦虑
  ↓ so what?
意味着续航是品类共性短板,不是单品牌问题
  ↓ so what?
品牌通过续航宣传难以差异化,需要换战场
  ↓ so what?
建议:把研发预算从"续航再加 20 分钟"转向"AI 自动剪辑闭环"
```

## 第三层 表达

### 3.1 有洞察、有观点(深度而非堆砌)

**核心**:发现要让读者"知道得比读之前多"。

约束:

- **每个发现必须有非显然性**——读者读到时会"咦,是这样吗 / 我没想过"。"用户喜欢便宜的"这种行业共识降级或删掉
- **三层穿透**:每个核心发现写到"用户说了什么(事实)→ 行为暴露了什么(模式)→ 这反映了什么(解读)"。只到事实层 = 信息;到解读层 = 洞察
- **观点要有立场**:对核心议题敢下判断("我们认为 X 比 Y 更可信,因为……")。不允许"两边都对"的中庸表述
- **避免堆砌**:同一论点下证据 3-5 条够了,超过 7 条说明没归纳清楚

自检:每个发现问——**如果读者只懂行业常识,他读完会觉得多学到了一件事吗?**没有 → 这个发现没价值。

### 3.2 易于理解

**核心**:无业务基础的读者也能跟上。

约束:

- **专业术语首次出现必须解释**(MECE / Cohen's Kappa / Fisher 精确检验 / NPS / JTBD 等),解释放脚注或括号
- **避免概念癌词组**——参照 [writing_style.md](writing_style.md):结构性 / 范式 / 锚点 / 底层 / 本质上 / 终极
- **不允许 "不是 X 而是 Y" 做标题**(口号化、削弱具体性)
- **复杂概念给具体例子**:抽象描述 ≤ 2 行,后面接一个用户原话或数据例子
- **句子平均长度 ≤ 30 字**,长句拆短

自检:让一个不懂这个行业的同事读核心结论,**3 分钟内能复述要点 = 过关**。

### 3.3 可视化与美观

**核心**:重点视觉化,排版降低阅读成本。

约束:

- **能用表格的不用文字段落**(对比、分群、量化数据)
- **每个核心发现配一个视觉锚点**:数字 / 表格 / 引语 / 对比图。纯文字段不超过 5 行
- **关键数字加粗**(占比、增长率、群体规模);避免全段加粗(等于没加粗)
- **引语单独成块**,带用户编号 + 一句场景描述,不混在正文
- **表格列数 ≤ 5**,超过拆表;**行数 > 8** 加分组或排序
- **章节内部层级 ≤ 3**(章 → 节 → 子节),再深读者会迷路

自检:**滚动浏览不读字**,能不能从视觉密度看出"哪里是重点、哪里是证据"?不能 → 排版没分层。

### 3.4 引人入胜

**核心**:节奏合理,信息密度适中,读者不弃读。

约束:

- **开头 200 字内完成 SCQA**,不允许"先讲行业历史"
- **核心结论章节 ≤ 1 页**,3 分钟拿到全部 takeaway
- **节奏控制**:数据段 → 引语段 → 解读段 交替,不连续 3 段同类
- **每个主题展开 ≤ 3 页**,超过拆子主题
- **不允许章末金句收尾**——以 "so what / 决策含义" 收尾,不以诗意收尾
- **信息密度**:每 200 字至少出现 1 个具体数据/原话/例子;否则属于"评论文风"

自检:连续读 3 章不疲劳 = 节奏对了;读到一半想跳过 = 信息密度不均。

## 优先级冲突时怎么办

冲突时取舍顺序:**底线 > 结构 > 表达**

- 底线和表达冲突 → 先保底线(数据严谨永远优先于好读)
- 结构和表达冲突 → 先保结构(金字塔的论点结构永远优先于叙事流畅度)
- 表达内部冲突 → 先保"易于理解"(看得懂 > 好看 > 有趣)

例:

- 为了"引人入胜"而把弱证据写成强结论 → 违反底线,不行
- 为了"可视化"而砍掉反面证据 → 违反底线,不行
- 为了"有洞察"而把 n=5 的群写成趋势 → 违反底线,不行
- 为了"美观"而把章节切成同样长度,导致 MECE 维度混乱 → 违反结构,不行

## 标准报告大纲

```
研究背景            [SCQA — 让读者知道为什么这个研究值得读]
  研究目标          [Q — 要回答的核心问题、要支持的业务决策]
  研究方法          [让读者校准证据强度]
  研究执行情况      [覆盖了什么、漏了什么、已知偏差]

核心结论            [A — 紧跟研究背景,在详细分析之前]
  中心论点          [1 句话,实型表达]
  关键论点 3-7 个   [每个一句话独立成立 = 详细分析的章节]
  关键决策建议      [对不同读者分组]

详细分析            [归纳论证 — 每个主题展开一个关键论点]
  主题 1            [按同一维度切分,横切维度作为交叉分析穿插]
  主题 2
  ...

待解问题与后续研究  [开放问题、建议补充的研究]

附录                [证据细节、样本清单、方法论补充、不在主线的有趣发现]
```

每节的具体写作约束见 [templates/simple_report.md](../templates/simple_report.md) 里的 inline 注释。

## 落笔后自检清单(13 项)

### 底线层(2)

- [ ] 每个数字都标了来源、能复现;n<10 的群没用百分比;宽窄口径分开标注
- [ ] 每个核心论点都对应研究目标里的某个问题;研究背景写明了要支持的业务决策

### 结构层(7)

- [ ] 开篇 200 字内完成 SCQA(情境-冲突-疑问-答案预告)
- [ ] 中心论点只有 1 个,关键论点 3-7 个
- [ ] 详细分析章节满足 MECE(同一切分维度,无重复)
- [ ] 章节标题都是论点句(实型),不是话题词
- [ ] 每个发现是"论点 + 证据 + 反面 + so what"四件套
- [ ] 每个发现都做过 so what 三连(到决策层)
- [ ] 章节顺序符合 Q&A 链(下层答上层问)

### 表达层(4)

- [ ] 每个发现包含非显然性,不是行业共识
- [ ] 专业术语首次出现都解释了;无概念癌词组;无 "不是 X 而是 Y" 标题
- [ ] 每段文字 ≤ 5 行;关键数据有视觉锚点;表格列数 ≤ 5
- [ ] 没有金句结尾;开头无冗长铺垫;信息密度均匀

任一项不过 → 改完再交付。
