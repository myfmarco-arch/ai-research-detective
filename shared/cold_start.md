# 冷启动:从空目录到 CONTEXT 就位

> 三个 skill(`research-archivist` / `research-detective` / `research-reviewer`)在没有 `CONTEXT.md` 的项目里启动时,共用本流程。
> 单一真源:本文件被各 SKILL.md 引用,改这里所有 skill 同步生效。

## 触发条件

- 当前目录没有 `CONTEXT.md`
- 或 `CONTEXT.md` 存在但速读卡 / 我的身份 / 研究问题任一字段空白或带 `<!-- 待用户补充 -->` 标记

## 原则:先扫再问,不问空白题

不要扔空模板让用户从零填。先扫项目,把能推断的字段填好,再请用户**一次性**补齐空白字段。

**确认优先于写入**:AI 可以生成 `CONTEXT.md` / `README.md` 草案,但草案只用于展示和讨论。未经用户明确确认,不得把草案写入最终文件,也不得把 AI 推断的研究问题、产出形式、底线当作已确认事实。

## 流程

### 步骤 1: 扫描项目,判定启动状态

先列出根目录、`data/`(如有)、其他可见目录的所有文件,再阅读关键文件内容(不只是文件名)。扫描后必须先完成两件事:启动状态分类 + 材料分层。

#### 1a. 启动状态分类

| 状态 | 识别信号 | 默认动作 |
| --- | --- | --- |
| **空目录** | 无研究文件,或只有系统/配置文件 | 不生成强假设;只生成最小 CONTEXT/README 草案,请用户补研究对象、问题、资料计划 |
| **裸资料态** | 有访谈、问卷、反馈、CSV、可用性记录、转写稿等一手资料,但无 wiki/报告 | 从资料推断研究对象和材料规模;建议后续走 archivist 入库 |
| **研究设计态** | 有研究计划、访谈提纲、招募标准、执行计划、会议计划 | 从设计文件推断研究问题、对象、样本边界;全部标为待用户确认 |
| **既有分析态** | 有报告、PPT、memo、结论草稿、复盘文档,但缺 wiki | 把它们标为二手分析/既有输出;询问用户是审查、继续分析、还是重建 wiki |
| **混合项目态** | 同时有原始资料、研究设计、旧报告/旧输出 | 先建立材料地图;明确一手资料、二手分析、研究设计、项目管理文件的边界 |
| **wiki 异常态** | 有 `wiki/` 但缺 `CONTEXT.md` | 不重建 wiki;只补 CONTEXT/README,并保留既有 wiki |

#### 1b. 材料分层

把扫描到的文件分成四层,写入 README 草案的材料地图:

- **一手资料**:访谈记录、问卷原始回答、用户反馈、可用性测试记录、原始 CSV、转写稿。可进入 `data/`,可作为 wiki 证据源。
- **二手分析**:旧报告、PPT、research memo、已有结论、策略文档。不能当作一手证据;只能作为既有观点、待验证假设、framework/benchmark 或审查对象。
- **研究设计**:研究计划、访谈提纲、招募标准、执行计划。用于补 CONTEXT/README,不直接作为用户证据。
- **项目管理**:时间表、会议纪要、任务列表、交付说明。用于理解约束和产出要求,不作为研究证据。

**红线**:旧报告/PPT/memo 不得混入一手证据层。若后续要从旧报告系统化回建 wiki,必须标注"二手分析来源",并优先要求用户补充或定位原始资料。

#### 1c. 可推断信息

从上述分类中提取:

- **研究对象**:文档里反复出现的产品名、品牌、人群标签
- **资料类型与规模**:一手资料 N 份 / 二手分析 N 份 / 研究设计 N 份 / 项目管理 N 份
- **可能的研究问题**:从研究设计、旧报告、需求文档中浮出的候选问题,全部标为"AI 推断,待用户确认"
- **材料边界与可信度**:哪些是一手证据,哪些只是旧判断或计划
- **参考资料**:现有方法论文档、行业报告、历史项目

空目录时不要编造这些字段;只标记为待用户补充。

### 步骤 2: 生成 CONTEXT/README 待确认草案

基于 [shared/templates/CONTEXT.md](templates/CONTEXT.md) 和 [shared/templates/README.md](templates/README.md):

- 推断到的字段直接填入草案,但标记为"AI 推断,待用户确认"
- 推断不到的字段保留空白,紧接其后插入 `<!-- 待用户补充 -->` 注释
- 不要删模板里的章节结构
- **不要写入最终 `CONTEXT.md` / `README.md`**;只在对话中展示草案或写入临时草稿文件(如 `process/context_draft.md`),等待用户确认

### 步骤 3: 一次性请用户补齐

把生成的初稿展示给用户,列出**所有标了"待用户补充"的字段**,请其一次性回答:

- 研究问题(核心 + 辅助)
- 汇报给谁
- 产出形式(报告 / 纪要 / 演示稿,大概长度)
- 产出位置(默认 `outputs/`,可覆盖)
- 1-3 条项目专属红线(不写通用方法学)
- 你在这个项目里的身份(专业训练 + 承担的角色)
- 项目特有的分析流程偏好(可选)

同时**请用户检查 AI 已填好的字段有没有错**——AI 推断可能错,用户校对一遍才安全。

### 步骤 4: 用户确认后合并写入

用户明确确认草案并补齐缺失字段后:

1. 把回答合并进初稿,写出最终 `CONTEXT.md` 和 `README.md`;如果用户只补充信息但未确认 AI 推断字段,先追问校对,不要落盘
2. **跑 `lint_context.py` 校验最终 CONTEXT.md**(红线 0 才算 cold_start 通过):

   ```bash
   python3 ${CLAUDE_SKILL_DIR}/../../shared/scripts/lint_context.py CONTEXT.md
   ```

   红线非 0(占位符残留 / 必填字段空 / 核心问题 < 20 字)→ 回到步骤 3 让用户继续补;黄线(底线套话 / 填充式动词)人工复核
3. 创建 `data/`(如不存在)、`process/`、`outputs/`
4. 如果根目录有未归类的研究资料,询问用户是否移入 `data/`(不要擅自移动)
5. **配置项目级硬约束(CLAUDE.md)**:
   - 检查项目根是否已有 `CLAUDE.md`
   - **没有**:告知用户"将从 skill 包复制 `shared/CLAUDE.md` 到项目根 `CLAUDE.md`(项目级硬约束的单一真源,各 SKILL.md 引用)",得到确认后执行 `cp ${CLAUDE_SKILL_DIR}/../../shared/CLAUDE.md <项目根>/CLAUDE.md`
   - **已有**(用户自己的项目级配置):**不要覆盖**。展示 skill 包 `shared/CLAUDE.md` 的内容,询问用户:① 追加到现有 `CLAUDE.md` 末尾,② 不动,我自己手动整合,③ 仍然覆盖(警告:会丢失现有内容)。默认推荐 ①
   - skill 包规则文件路径:本 plugin 内的 `shared/CLAUDE.md`,即从当前 SKILL.md 文件位置上溯两级(`skills/research-detective/SKILL.md` → `skills/` → plugin 根)再进入 `shared/`

### 步骤 5: 确认下一步意图

cold-start 结束前不要自动进入某个 skill 的主体流程。先把扫描出的启动状态、材料分层、待确认字段和建议下一步展示给用户,让用户确认要做什么:

| 用户下一步意图 | 进入路径 |
| --- | --- |
| 建 wiki / 入库资料 / 系统化整理原始资料 | `research-archivist` 常规流程 |
| 基于资料或 wiki 分析、综合、写报告、修订结论 | `research-detective` 常规流程 |
| 审查已有报告、找反证、检查 unsupported claims | `research-reviewer` 常规流程;若没有 CONTEXT,按 reviewer 的轻量审查分支处理 |
| 只初始化项目结构 | 停在 CONTEXT/README/目录创建,不继续分析或入库 |
| 从旧报告反推资料地图或重建 wiki | 先把旧报告标为二手分析/待验证假设,再询问是否能补充原始资料;没有原始资料时不得把旧报告当一手证据 |

用户确认下一步前,不得擅自移动文件、入库、分析或审查。

### 步骤 6: 进入各 skill 的常规流程

- `research-archivist` 进入资料评估和入库
- `research-detective` 进入证据采集和分析
- `research-reviewer` 进入对抗性审查

## 增量场景的处理

当 `CONTEXT.md` 已存在但部分字段缺失:

- 不重新扫项目,只针对缺失字段问用户
- 不动用户已填的字段,即使 AI 觉得不准

## 不做的事

- **不做的事 1**: 不读 CONTEXT 直接开始分析。这等于默认违反视角约束
- **不做的事 2**: 不一字段一问。一次性把空白字段列全,用户一次回答
- **不做的事 3**: 不擅自把根目录文件移入 `data/`。问过用户再动
